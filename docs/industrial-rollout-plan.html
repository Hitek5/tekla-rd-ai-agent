Пошаговый план внедрения Tekla/RD AI Agent

Дорожная карта от нулевой инициативы до промышленной эксплуатации локального ИИ-агента для Tekla/RD в закрытом корпоративном контуре. Документ ориентирован на управленческое согласование, запуск пилота, подготовку инфраструктуры, контроль безопасности и передачу решения в эксплуатацию.

1. Принципы внедрения

Сначала управляемость, потом автономность

На первых этапах агент не изменяет производственные модели самостоятельно. Любое изменение проходит через dry-run, approval и журналирование.

Typed tools вместо произвольного кода

LLM не получает прямой канал выполнения C#. Повторяемые успешные действия оформляются как типизированные инструменты Tekla.

Промышленность через evidence pack

Каждый gate закрывается набором артефактов: архитектура, протоколы тестов, акты приемки, журнал рисков, инструкции эксплуатации.

Рабочее допущение: внедрение идет в среде с формальными согласованиями, разделением ролей, внутренними контурами поставки и обязательной проверкой изменений перед вводом в эксплуатацию.

Проектный офис Календарь, протоколы, контроль gates, риск-реестр, синхронизация подразделений. План-график, протоколы совещаний, RAID log.

3. Сводный календарный план

Недели 0-2
Инициирование. Назначить владельцев, оформить паспорт инициативы, определить пилотную область, создать рабочую группу.
Недели 2-5
Требования и границы. Зафиксировать use cases, данные, ограничения, критерии успеха, роли и перечень артефактов для согласования.
Недели 5-8
Архитектура и контроль рисков. Подготовить HLD/LLD, threat/risk review, схему контуров, policy для инструментов и audit.
Недели 7-12
Инфраструктура. Подготовить GPU-сервер, зеркала зависимостей, модельный каталог, TLS, мониторинг и backup.
Недели 10-16
MVP без production Tekla. Запустить orchestrator, RAG, eval, dry-run tool gateway и первичный контур аудита.
Недели 14-20
Интеграция с рабочим местом. Подключить Tekla Open API adapter, read-only tools, затем approved create/modify на копиях моделей.
Недели 20-28
Пилот. 3-5 пользователей, фиксированные сценарии, замер качества, журнал дефектов, корректировка регламентов.
Недели 28-36
Промышленный запуск. Приемка, эксплуатационные инструкции, SLA/SLO, обучение, change process, post-launch monitoring.

Сроки являются ориентировочными. При необходимости внешних закупок, дополнительных проверок или расширенного контура испытаний календарь следует вести как gated-план, а не как жесткий спринтовый график.

4. Этап 0. Инициирование

Недели 0-2
Веха Ответственный Что сделать Результат / критерий готовности
0.1 Паспорт инициативы Бизнес-заказчик, проектный офис Описать цель, проблему, ожидаемый эффект, границы MVP, предварительные риски, зависимые подразделения. Паспорт инициативы согласован; назначен спонсор и владелец продукта.
0.2 Рабочая группа Проектный офис Назначить представителей CAD/Tekla, ИТ, эксплуатации, безопасности, QA, data owner и разработчиков. Утверждена RACI-матрица и регулярный ритм встреч.
0.3 Пилотная область Владелец Tekla/CAD-процесса Выбрать подразделение, типовые модели, первые 3-5 сценариев, перечень ограничений для пилота. Зафиксирован список pilot use cases и ответственные инженеры-проверяющие.
0.4 Реестр решений и рисков Технический руководитель, проектный офис Открыть ADR/RAID log: архитектурные решения, риски, допущения, открытые вопросы. Все спорные решения фиксируются письменно и имеют владельца.
Gate 0: есть спонсор, рабочая группа, первичный backlog, пилотная область и разрешение переходить к детализации требований.

5. Этап 1. Требования и границы

Недели 2-5
Веха Ответственный Что сделать Результат / критерий готовности
1.1 Каталог сценариев Владелец продукта, CAD owner Разложить сценарии по категориям: чтение модели, создание элементов, армирование, чертежи, проверка, объяснение API. Каталог сценариев с приоритетом, сложностью, риском и критериями приемки.
1.2 Data inventory Data owner Описать источники: Tekla examples, внутренний C# код, шаблоны РД, справка, тестовые модели, будущие pilot logs. Реестр источников с владельцем, статусом допуска, местом хранения и сроком жизни.
1.3 Нефункциональные требования Технический руководитель, эксплуатация Зафиксировать требования к доступности, времени ответа, журналированию, хранению артефактов, backup/restore, обновлениям. NFR-документ согласован с инфраструктурой и эксплуатацией.
1.4 Границы автоматизации Бизнес-заказчик, CAD owner Определить, что агент может делать сам, что только в dry-run, что требует ручной приемки, что исключено из MVP. Матрица разрешенных/запрещенных действий утверждена и перенесена в policy backlog.
1.5 Метрики успеха Владелец продукта, QA Определить baseline: tool-call validity, human correction time, compile rate, CAD acceptance rate, audit coverage. Метрики и способ измерения добавлены в eval plan.
Gate 1: use cases, данные, NFR и границы автоматизации согласованы; можно проектировать архитектуру и контур контроля.

6. Этап 2. Архитектура и контроль рисков

Недели 5-8
АрхитектураHLD, LLD, deployment boundaries, network flows.
БезопасностьRisk review, policy gates, журналирование, supply chain controls.
ДанныеКлассификация источников, правила обезличивания, статусы review.
РазработкаTool contracts, API boundary, тестовый контур.
ЭксплуатацияMonitoring, backup, incident flow, change model.
QAAcceptance tests, regression set, offline checks.
Веха Ответственный Что сделать Результат / критерий готовности
2.1 HLD/LLD Технический руководитель Описать GPU server, orchestrator, RAG, model serving, workstation host, audit, зеркала зависимостей. Архитектурный пакет согласован рабочей группой.
2.2 Tool policy design Технический руководитель, CAD owner Определить read-only, simulation, create, modify, delete, export. Для каждого класса указать approval и ограничения. Policy перенесена в `configs/tools-policy.yaml` и покрыта тестами.
2.3 Security review Информационная безопасность Проверить архитектуру, границы trust zones, запрет произвольного кода, supply chain, audit, хранение данных. Получены замечания, назначены владельцы устранения, зафиксирован минимальный набор мер.
2.4 Модель эксплуатации Эксплуатация Описать владельца сервиса, каналы обращений, график обновлений, rollback, резервное копирование, incident flow. Черновик operational runbook принят в работу.
2.5 План приемочных испытаний QA Составить test matrix: offline acceptance, функциональные CAD-сценарии, safety scenarios, model eval, pilot acceptance. Тест-план привязан к gates и метрикам успеха.
Gate 2: архитектура согласована условно или полностью; критичные замечания внесены в backlog до запуска MVP.

7. Этап 3. Инфраструктура и поставка зависимостей

Недели 7-12
Веха Ответственный Что сделать Результат / критерий готовности
3.1 GPU server readiness Инфраструктура Подготовить ОС, NVIDIA driver, Docker, storage layout, сетевые доступы, учетные записи администрирования. `nvidia-smi`, Docker GPU smoke test, baseline monitoring работают.
3.2 Internal mirrors Инфраструктура, безопасность Развернуть или подключить внутренние источники Docker/OCI, PyPI, NuGet, Git, model storage. Установка пакетов и pull образов возможны без внешнего доступа.
3.3 Air-gap bundle DevOps / инфраструктура Собрать bundle: images, wheels, NuGet, git bundles, model manifest, SHA256SUMS. Bundle проверен, импортирован и зарегистрирован во внутреннем хранилище.
3.4 Model catalog Технический руководитель, data owner Зафиксировать модели, лицензии, хеши, источник, назначение, профиль железа, ограничения использования. Модельный каталог утвержден для MVP; запрещены неучтенные веса.
3.5 TLS and access model Инфраструктура, безопасность Настроить внутренний TLS, DNS/service name, минимально необходимые доступы, журналы доступа. Доступы выданы по ролям; нет неучтенных открытых endpoint.
Gate 3: инфраструктура готова к MVP; зависимости устанавливаются из внутренних источников; внешний доступ не требуется для штатного запуска.

8. Этап 4. MVP без Tekla production

Недели 10-16
Веха Ответственный Что сделать Результат / критерий готовности
4.1 Orchestrator Разработка Запустить FastAPI orchestrator, health checks, LLM endpoint, RAG lookup, audit events. Чат работает на тестовом корпусе; ошибки модели и RAG журналируются.
4.2 Initial RAG corpus Data owner, CAD owner Подготовить Tekla docs/examples, внутренние C# примеры, шаблоны РД, статусы verification. Корпус индексируется; deprecated/blocked источники исключаются из retrieval.
4.3 Eval harness QA, разработка Собрать 100-300 задач: read, create dry-run, unsafe prompts, объяснения API, drawing draft. Есть baseline base model vs RAG; результаты сохраняются в evidence pack.
4.4 Policy tests Разработка, QA Покрыть unit tests: unknown tool blocked, mutation needs approval, production write blocked, read allowed. Policy tests входят в обязательную проверку перед release.
4.5 Demo review Владелец продукта Провести демонстрацию без подключения к production Tekla: чат, RAG, dry-run, audit, unsafe prompt. Рабочая группа подтверждает, что MVP можно подключать к тестовому рабочему месту.
Gate 4: MVP доказал базовую полезность и управляемость без риска для производственных моделей.

9. Этап 5. Интеграция с Tekla workstation

Недели 14-20
Веха Ответственный Что сделать Результат / критерий готовности
5.1 Workstation host hardening Разработка, безопасность Закрепить bind на localhost, добавить approval header validation, локальный audit, deny-by-default routing. Обход orchestrator не позволяет выполнить mutating tool без локального контроля.
5.2 Tekla Open API adapter Разработка, CAD owner Заменить stub facade на adapter Tekla API для read-only инструментов; затем создать отдельные адаптеры для create tools. GetSelection, QueryObjects, ValidateModel работают на тестовой модели.
5.3 Tool acceptance CAD owner, QA Для каждого инструмента подготовить test model, expected result, rollback/compensating action, список предупреждений. Инструмент получает статус experimental, pilot-ready или blocked.
5.4 Approval UX Владелец продукта, разработка Сделать понятный сценарий подтверждения: что будет изменено, где, какие объекты, как откатить. Инженер видит dry-run diff/summary перед подтверждением.
5.5 Workstation package Инфраструктура, эксплуатация Подготовить установочный пакет, инструкцию, версионирование, журнал обновлений, способ отключения. Хост устанавливается и удаляется регламентно; версия отслеживается.
Gate 5: read-only tools работают на рабочем месте; mutating tools доступны только на копиях моделей и только через approval.

10. Этап 6. Пилот

Недели 20-28
Веха Ответственный Что сделать Результат / критерий готовности
6.1 Pilot charter Бизнес-заказчик, владелец продукта Утвердить 3-5 пользователей, список моделей-копий, сценарии, расписание, критерии прекращения пилота. Все участники понимают рамки пилота и не используют production-модели для автономных изменений.
6.2 Training session Владелец продукта, CAD owner Обучить пользователей: как писать запросы, читать dry-run, подтверждать действия, сообщать дефекты. Пользователи прошли вводный инструктаж; есть канал поддержки.
6.3 Pilot execution CAD owner, QA Выполнять сценарии по плану, фиксировать результаты, корректировки инженера, время, ошибки и отклонения. Накоплена статистика по каждому сценарию; audit покрывает 100% tool attempts.
6.4 Weekly triage Проектный офис, технический руководитель Разбирать дефекты, prompt failures, RAG gaps, policy false positives/negatives, инфраструктурные проблемы. Каждая проблема имеет владельца, severity, срок и решение: fix, defer, accept, block.
6.5 Pilot report QA, владелец продукта Собрать отчет: метрики, полезность, риски, дефекты, рекомендации, готовность к промышленному запуску. Приемочная комиссия принимает решение: go, go with conditions или no-go.
Gate 6: pilot report подтверждает ценность, безопасность процесса и готовность эксплуатационной модели.

11. Этап 7. Приемка и промышленный запуск

Недели 28-36
Веха Ответственный Что сделать Результат / критерий готовности
7.1 Evidence pack Проектный офис, QA Собрать полный пакет: архитектура, тесты, pilot report, risk register, инструкции, журнал изменений, известные ограничения. Пакет готов для формальной приемки и передачи в эксплуатацию.
7.2 Operational readiness review Эксплуатация Проверить мониторинг, alerting, backup/restore, обновления, incident flow, доступы, capacity plan. Эксплуатация подтверждает готовность принять сервис.
7.3 Final security review Информационная безопасность Проверить устранение замечаний, логи, доступы, supply chain, запрет production writes без регламента. Нет критичных открытых замечаний; условия запуска документированы.
7.4 Change approval Проектный офис, эксплуатация Оформить изменение: дата запуска, план отката, окно работ, ответственные, коммуникация пользователям. Изменение утверждено; rollback план проверен.
7.5 Production launch Эксплуатация, технический руководитель Ввести сервис в промышленный контур в ограниченном режиме, включить усиленный мониторинг и daily check. Сервис доступен целевой группе; нет критичных инцидентов в stabilization window.
Gate 7: решение принято в промышленную эксплуатацию с ограничениями, SLA/SLO и назначенными владельцами.

12. Этап 8. Эксплуатация и развитие

После запуска
Направление Ответственный Регулярные действия Контрольный результат
Service operations Эксплуатация Мониторинг доступности, latency, GPU utilization, ошибок LLM, ошибок workstation host. Еженедельный service report; инциденты закрываются по SLA.
Model and RAG quality Владелец продукта, QA Обновление eval set, анализ пользовательских корректировок, review новых источников RAG. Качество не деградирует после обновлений; regression suite проходит.
Change management Технический руководитель Любые новые tools, модели, prompts, RAG sources проходят review, тесты и approval. Нет незарегистрированных изменений в production.
Fine-tuning readiness Data owner, технический руководитель Собирать verified SFT dataset из успешных сценариев, не тренировать на raw logs без review. LoRA/QLoRA запускается только после baseline и data quality gate.
Knowledge transfer Владелец продукта, эксплуатация Обновлять инструкции, FAQ, шаблоны запросов, список ограничений, обучающие материалы. Новые пользователи могут подключиться без участия разработчиков.

14. Контрольные gates и обязательные артефакты

Gate Решение Минимальные артефакты Условие перехода
Gate 0 Старт проработки Паспорт инициативы, рабочая группа, pilot use cases, RAID log. Есть спонсор, владелец продукта и согласованная область MVP.
Gate 1 Переход к архитектуре Каталог сценариев, data inventory, NFR, метрики успеха. Границы автоматизации понятны и приняты владельцами процесса.
Gate 2 Разрешение на MVP build HLD/LLD, security review, tool policy, test plan, operational draft. Критичные замечания имеют план устранения; нет блокирующих рисков.
Gate 3 Готовность инфраструктуры GPU smoke test, internal mirrors, model catalog, access model, backup draft. Сервис можно запускать без внешнего доступа и без ручных обходов поставки.
Gate 4 Готовность MVP Orchestrator, RAG corpus, eval report, policy tests, audit sample. Dry-run сценарии и unsafe prompts проходят контрольные проверки.
Gate 5 Готовность Tekla pilot Workstation host, Tekla adapter, tool acceptance, install guide. Read-only tools работают; mutating tools ограничены approval и копиями моделей.
Gate 6 Решение по промышленному запуску Pilot report, defect register, risk closure, user feedback, service model. Приемочная комиссия принимает go / go with conditions.
Gate 7 Передача в эксплуатацию Evidence pack, operational runbook, final review, change approval, rollback plan. Эксплуатация принимает сервис; назначены SLA/SLO и владельцы.
Стоп-факторы перед промышленным запуском: нет владельца сервиса, не работает audit, есть неучтенные модели или зависимости, mutating tools обходят approval, нет rollback-плана, pilot report не подтверждает качество на ключевых сценариях.

15. Минимальный пакет документов к промышленной эксплуатации

Управление и приемка

  • Паспорт инициативы и решение о запуске.
  • Каталог use cases и критерии приемки.
  • RACI, RAID log и протоколы gates.
  • Pilot report и решение приемочной комиссии.
  • Release notes и план коммуникации пользователям.

Технология и эксплуатация

  • HLD/LLD и deployment diagram.
  • Security review и перечень мер контроля.
  • Operational runbook, backup/restore, incident flow.
  • Model catalog, dependency manifest, checksums.
  • Test reports, regression suite, audit samples.

Документ можно распечатать или сохранить в PDF из браузера. Для рабочей версии допускается добавить внутренние номера заявок, протоколов и согласующих комитетов.
